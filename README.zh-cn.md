# brkpt-auth

透明、可组合、可移植的 NestJS 六边形身份认证方案

---

## 设计理念

主流的身份认证方案一般分为三类：

**托管服务**：提供 API，认证逻辑与用户数据完全托管在供应商一侧。集成成本低，但数据主权让渡，且容易陷入供应商锁定（Vendor Lock-in）。此外，数据库结构、认证状态的管理方式（传统 session、无状态 JWT 还是有状态 JWT）、token 的传输方式（cookie 还是 header）、token 字段定义等核心设计均由供应商预先决定，项目只能围绕其设计思路构建。由于实现细节完全不透明，定制空间极为有限。可以把它看作云端的黑盒。

**自托管库**：提供 npm 包，部署在你自己的基础设施上，数据主权得以保留。但与托管服务存在同样的根本问题：核心设计固定且高度抽象，项目依然要围绕其设计思路构建，定制需要先理解其内部心智模型，且受限于内部结构，超出其设计范围的需求往往难以实现，插件化方案也效果有限。代码虽然以 node_modules 的形式存在于本地，但内部实现依然复杂，难以修改维护。可以把它看作本地的黑盒。

**样板代码**：提供完整源码，代码归你所有，但往往固执己见。它针对特定的用户模型、数据库和项目结构编写，有强烈的设计偏见，甚至内置了一系列固定配置，与你的项目需求不符的地方都需要自行修改。尽管大多数样板代码声称“开箱即用、架构优雅、易于扩展、完全生产就绪”，但实际情况往往相去甚远：耦合严重、架构边界模糊、结构冗余、可读性差、不符合最佳实践、功能缺失、难以调试的漏洞、难以二次开发。即使在其基础上完成了项目，下一个新项目依然要重新经历一遍同样的流程。

除此之外，还有一种最常见的做法，那就是不依赖任何服务或框架，完全自行实现。这种做法完全可控，但代价同样明显：缺少统一的约定和规范，难以评估实现是否符合最佳实践，逻辑容易散落各处，部分功能难以实现，新增需求可能引发大范围架构调整，且每个新项目都要重复同样的工作。

brkpt-auth提供了一种不同的做法：

- **透明** — 完整源码直接安装到你的项目里，没有编译产物或隐藏行为，结构清晰可读，可以轻易修改。
- **可组合** — 按需添加功能，各功能相互独立，通过事件发射器松耦合通信。
- **非侵入** — 不假设你的数据库结构、用户模型或 JWT 字段定义。通过端口-适配器模式，只需实现对应接口，即可将 brkpt-auth 接入你的项目，无需为此调整现有代码。
- **可移植** — 业务逻辑与适配器分离，换项目时直接带走，无需重写核心逻辑，只需为新项目重写适配器来接入。
- **NestJS 原生** — 从头为 NestJS 构建，原生使用模块、守卫、装饰器与事件发射器，无兼容层。
- **六边形架构** — 服务层持有固定的业务逻辑，端口定义接口，适配器由你实现，边界始终清晰。

使用很简单：通过 `brkpt-cli` 初始化, 实现你的适配器, 在 `AppModule` 中注册。每个适配器只需实现几个类型化的方法，通常就是直接调用你项目中已有的实现。

## 功能列表

| 功能              | 说明                                 |
| ----------------- | ------------------------------------ |
| `core`            | 无状态 JWT 双令牌认证。**必需项**。  |
| `credentials`     | 账号密码注册与登录                   |
| `oauth`           | 社交账号认证                         |
| `otp`             | 一次性密码认证与验证                 |
| `magic-link`      | 魔法链接认证与验证                   |
| `session`         | 基于无状态 JWT 的会话管理层          |
| `blacklist`       | 实时访问令牌吊销                     |
| `verify-email`    | 邮箱验证门控，未验证账号在验证前受限 |
| `change-password` | 已认证状态下的密码修改               |
| `reset-password`  | 通过身份证明重置密码                 |
| `audit`           | 用户自定义的关键认证事件处理器       |

## 仓库结构

```plain
brkpt-auth/
 ┣ demo/     # 完整可运行的 NestJS 示例项目
 ┗ lib/      # 由 brkpt-cli 分发的源码模板
```

### 运行示例

将 `.env.example` 复制为 `.env` 并填入配置，然后：

```bash
cd demo
pnpm install
pnpm start:dev
```

示例项目使用内存用户存储，但需要本地 Redis 实例。E2E 测试使用内存模拟来替代 Redis 和邮件，无需外部服务。

```bash
pnpm test        # unit tests
pnpm test:e2e    # e2e tests
```
